Demo Control Panel — How It Works

A Playwright-driven live-demo runner for the NutriBase app. It opens real, headed Chromium windows — one per role — and clicks them through a scripted clinical demo so a presenter can drive it live or record it, instead of improvising through the real app on stage.

Extracted from nutri-ledger/_docs_uni/presentation/demo-runner into its own repository so it can evolve independently of the app monorepo.
admin doctor patient
01 · Architecture

Three persistent Chromium instances — one per role, since each needs its own login session — are owned by a small Express server. Two front ends trigger the same shared step definitions against those instances: a clickable web panel, or a terminal script that runs everything in order.

                              │ drives via Playwright

Not self-hosting. This repo only drives a browser against NutriBase — it never serves the app itself. NutriBase's frontend and backend must already be running at BASE_URL before you start the panel or the terminal script.

BASE_URL is set from the env var at startup but is a mutable, exported let in steps.ts — the panel's header bar can repoint it at any environment at runtime (e.g. a Railway-hosted deploy instead of localhost:3000) via POST /api/base-url, with no server restart. See §09.

02 · The two front ends
Entry pointCommandBehavior
server.ts npm run panel Web UI at :4949. Every step is an independent button — click any step on any instance in any order. Built for live presenting: recover from a skipped step without restarting.
demo.ts npm run demo Terminal fallback. Runs every step from steps.ts in fixed order, printing the talking point after each one and waiting for Enter before advancing.

Both are thin — all step logic lives once, in steps.ts. Neither front end contains any Playwright actions itself; they only call step.run() against whichever instance the step belongs to.

03 · Running one step

Clicking Run on a card in the panel walks through the same path every step takes, in the web UI:

request sequence — POST /api/steps/fatima-profile/run
you (browser)        server.ts                      Chromium page
     │                          │                              │
     │── click "Run" ──────────►│                              │
     │                          │── ensureInstance('doctor') ─►│
     │                          │   (reuses the open page, or          │
     │                          │    launches maximized Chromium)       │
     │                          │                              │
     │                          │── step.run({page, state, ──►│  click / fill / goto
     │                          │    checkpoint})               │  against NutriBase
     │                          │                              │
     │                          │◄── Partial<DemoState> ──────│
     │                          │   merged only if run()            │
     │                          │   resolved without throwing       │
     │◄── { ok: true, url } ────│                              │
     │  card turns green         │                              │

A step that throws mid-way — a selector that doesn't match, a timeout — responds 500 and its card turns red with the error message, but the instance's state is left exactly as it was: the patch is only applied with Object.assign(instance.state, patch) after run() resolves successfully, never partially.

04 · Shared state per instance

Each role instance carries its own small DemoState — currently just two captured profile URLs, filled in by earlier steps and read by later ones on the same instance:

steps.ts
interface DemoState {
  fatimaUrl: string   // set by "Patient List → Fatima Hadžić"
  stefanUrl: string   // set by "Stefan Jovanović — Intake Flags"
}

This is why some cards say "Requires: run X first" — it's a readable hint in the UI, not an enforced lock. Running a step out of order that reads an unset URL fails fast with a clear message ("Fatima's profile URL not captured yet — run that patient's list/lookup step first.") rather than clicking blind.

05 · Step-through mode

Every atomic Playwright action inside a step's run() is wrapped in act(label, fn) instead of being called directly. Each instance has an independent Arm step-through toggle; while armed, act() blocks after its action completes and stays blocked until the presenter clicks Continue → — which releases exactly that one action and re-arms for the next act() call. Arming it once therefore stops you at every logical action in a step, in sequence — open a page, type a search, press enter, click a result — not just the first breakpoint you happen to catch.

act(label, fn) — server.ts
result = await fn()                     // run the actual Playwright action
if (instance.paused) {              // step-through armed on this instance?
  instance.waitingLabel = label
  await new Promise(resolve => {   // blocks here — no polling loop
    instance.resumeResolver = resolve
  })
}
return result

Continue (POST /instances/:id/continue) calls resumeResolver() for just the current action — the paused flag itself is untouched, so the very next act() call blocks again. Toggling Arm step-through off mid-block instead calls resumeResolver() and clears paused, releasing the rest of the step to run unattended. This replaces an earlier design where a single boolean pause flag was checked by a polling loop — that version only froze at whichever checkpoint you caught before the flag flipped back off, then ran the remaining checkpoints in the same step at full speed with no further pausing.

06 · Telling instances apart

Once you have three maximized Chromium windows spread across two monitors, tagInstanceWindow() injects a colored inset frame and a corner badge into every page each instance loads — red for admin, blue for doctor, green for patient — via page.addInitScript(). A MutationObserver reinstalls it if a client-side route change ever strips it.

Gotcha — already fixed, don't reintroduce it. That init script is passed as a raw string, not a function. Playwright normally accepts a function and serializes it via .toString() — but tsx's esbuild transform injects a __name(...) helper call into named functions/arrows for stack-trace preservation, and that helper only exists in the Node runtime. Once injected into the browser it throws immediately and silently, before the frame or badge ever appears.
07 · Rehearsing safely

Two steps have real side effects: Create Visit + Vitals Live mutates real patient data, and the AI Diet Plan step makes an actual OpenAI API call. Setting DRY_RUN=1 makes both steps short-circuit right after navigating to the right tab — everything else in the demo runs normally.

rehearsal vs. the real thing
$ DRY_RUN=1 npm run panel     # rehearse — no data written, no API cost
$ npm run panel               # the real defense, or a deliberate re-seed
08 · Not built yet
  • Video recording. Playwright's recordVideo would fit a "shoot small takes, redo just the broken one" workflow — each page becomes a take, and a future New Take action would close the page (finalizing its .webm) and open a fresh one in the same context, so cookies and login survive between takes.
  • Secondary-monitor auto-positioning. Deliberately skipped — instances launch maximized wherever Windows opens them, and get dragged to the second monitor by hand.
09 · Change log

Every subsequent change to this project gets an entry here — what changed, why, and how — so the doc stays a running trace instead of a one-time snapshot.

2026-07-03 Runtime-switchable base URL (demo against Railway, not just localhost)
What
Added a base-URL bar to the panel header (text input + Set + a Local quick preset) and GET/POST /api/base-url in server.ts. Switching the URL closes and clears every open role instance, since a browser already logged in against the old environment would otherwise sit there silently pointed at the wrong app.
Why
The panel could previously only target one app instance for its whole process lifetime, fixed by the BASE_URL env var at startup. The same NutriBase app is also hosted on Railway, and the presenter needed to demo against that hosted environment without restarting the panel process (and losing whatever instances/state were already set up) each time they switch.
How
steps.ts: changed export const BASE_URL to export let BASE_URL and added setBaseUrl(url), which trims/strips a trailing slash and rejects anything not starting with http(s)://. Every step still reads the module-level BASE_URL binding by name at call time, and ES module bindings are live, so server.ts/demo.ts pick up the change automatically — no need to thread the URL through StepContext. server.ts's POST /api/base-url calls setBaseUrl() then closes every entry in the instances map. public/app.js loads the current value on init and posts changes on Set/Enter/Local.
2026-07-03 Per-action step-through, replacing the single-toggle pause
What
Replaced checkpoint() with act(label, fn) in every step. Added a one-shot Continue endpoint/button distinct from the Arm step-through toggle; the panel now shows the label of whatever action it's currently waiting after.
Why
Steps only had 2–3 hand-placed checkpoints each, so pausing landed on coarse boundaries covering several actions at once. Worse, the old boolean-toggle pause didn't re-arm itself — resuming once let the rest of the step's checkpoints fly through unattended. Wanted a real step-through: freeze after every logical action, every time, until explicitly told to continue.
How
Rewrote every step in steps.ts to wrap each Playwright action in act('label', fn). In server.ts, act() blocks on a per-call Promise (no polling) when instance.paused is true; a new /continue route resolves just that one block without touching the paused flag, so the next act() blocks again automatically. Disarming step-through mid-block now releases the current wait immediately instead of leaving it frozen. Updated public/app.js/index.html to show a live "waiting after: …" label and a Continue → button, and dropped the panel's poll interval from 4s to 1.5s for snappier feedback.
