Demo Control Panel — How It Works

A Playwright-driven live-demo runner for the NutriBase app. It opens real, headed Chromium windows — one per role — and clicks them through a scripted clinical demo so a presenter can drive it live or record it, instead of improvising through the real app on stage.

Extracted from nutri-ledger/_docs_uni/presentation/demo-runner into its own repository so it can evolve independently of the app monorepo.
admin doctor patient
01 · Architecture

Three persistent Chromium instances — one per role, since each needs its own login session — are owned by a small Express server. Two front ends trigger the same shared step definitions against those instances: a clickable web panel, or a terminal script that runs everything in order.

Not self-hosting. This repo only drives a browser against NutriBase — it never serves the app itself. NutriBase's frontend and backend must already be running at BASE_URL before you start the panel or the terminal script.
02 · The two front ends
Entry pointCommandBehavior
server.ts npm run panel Web UI at :4949. Every step is an independent button — click any step on any instance in any order. Built for live presenting: recover from a skipped step without restarting.
demo.ts npm run demo Terminal fallback. Runs every step from steps.ts in fixed order, printing the talking point after each one and waiting for Enter before advancing.

Both are thin — all step logic lives once, in steps.ts. Neither front end contains any Playwright actions itself; they only call step.run() against whichever instance the step belongs to.

03 · Running one step

Clicking Run on a card in the panel walks through the same path every step takes, in the web UI:

request sequence — POST /api/steps/fatima-profile/run
you (browser)        server.ts                      Chromium page
     │                          │                              │
     │── click "Run" ──────────►│                              │
     │                          │── ensureInstance('doctor') ─►│
     │                          │   (reuses the open page, or          │
     │                          │    launches maximized Chromium)       │
     │                          │                              │
     │                          │── step.run({page, state, ──►│  click / fill / goto
     │                          │    checkpoint})               │  against NutriBase
     │                          │                              │
     │                          │◄── Partial<DemoState> ──────│
     │                          │   merged only if run()            │
     │                          │   resolved without throwing       │
     │◄── { ok: true, url } ────│                              │
     │  card turns green         │                              │

A step that throws mid-way — a selector that doesn't match, a timeout — responds 500 and its card turns red with the error message, but the instance's state is left exactly as it was: the patch is only applied with Object.assign(instance.state, patch) after run() resolves successfully, never partially.

04 · Shared state per instance

Each role instance carries its own small DemoState — currently just two captured profile URLs, filled in by earlier steps and read by later ones on the same instance:

steps.ts
interface DemoState {
  fatimaUrl: string   // set by "Patient List → Fatima Hadžić"
  stefanUrl: string   // set by "Stefan Jovanović — Intake Flags"
}

This is why some cards say "Requires: run X first" — it's a readable hint in the UI, not an enforced lock. Running a step out of order that reads an unset URL fails fast with a clear message ("Fatima's profile URL not captured yet — run that patient's list/lookup step first.") rather than clicking blind.

05 · Pause & checkpoints

Each instance has an independent Pause toggle. A handful of steps call checkpoint() at natural narrative breaks — after a search demo, before a dialog submit — so execution can be frozen there mid-step to catch your breath, without rebuilding every action into an individually resumable micro-step.

step running ──► checkpoint() ──► paused? ──yes──► poll every 200ms

06 · Telling instances apart

Once you have three maximized Chromium windows spread across two monitors, tagInstanceWindow() injects a colored inset frame and a corner badge into every page each instance loads — red for admin, blue for doctor, green for patient — via page.addInitScript(). A MutationObserver reinstalls it if a client-side route change ever strips it.

Gotcha — already fixed, don't reintroduce it. That init script is passed as a raw string, not a function. Playwright normally accepts a function and serializes it via .toString() — but tsx's esbuild transform injects a __name(...) helper call into named functions/arrows for stack-trace preservation, and that helper only exists in the Node runtime. Once injected into the browser it throws immediately and silently, before the frame or badge ever appears.
07 · Rehearsing safely

Two steps have real side effects: Create Visit + Vitals Live mutates real patient data, and the AI Diet Plan step makes an actual OpenAI API call. Setting DRY_RUN=1 makes both steps short-circuit right after navigating to the right tab — everything else in the demo runs normally.

rehearsal vs. the real thing
$ DRY_RUN=1 npm run panel     # rehearse — no data written, no API cost
$ npm run panel               # the real defense, or a deliberate re-seed
08 · Not built yet
  • Video recording. Playwright's recordVideo would fit a "shoot small takes, redo just the broken one" workflow — each page becomes a take, and a future New Take action would close the page (finalizing its .webm) and open a fresh one in the same context, so cookies and login survive between takes.
  • Secondary-monitor auto-positioning. Deliberately skipped — instances launch maximized wherever Windows opens them, and get dragged to the second monitor by hand.
